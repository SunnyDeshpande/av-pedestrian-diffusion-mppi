possible associations (topics / pipelines)
==========================================

perception fuse → predictor → controller (adapt_prediction_launch patterns)
--------------------------------------------------------------------------
  lidar_processing
      └ publishes /lidar_pedestrian_position
  rgbd pedestrian detector (when fixed/ros2-run)
      └ publishes /rgbd_pedestrian_position

  lidar_camera_fusion
      └ subscribes /lidar_pedestrian_position & /rgbd_pedestrian_position
      └ publishes /fusion_pedestrian_position

  pedestrian_behaviour_predictor  OR  diffusion infer_node     (alternate; launch picks one mode)
      └ typically consume /fusion_pedestrian_position (and vehicle speed msgs)
      └ publish pedestrian_motion, pedestrian_ttc, person_prediction(+markers), diffusion adds pedestrian_predictions_tensor

  adapt_mppi_node
      └ GNSS (/navsatfix) + INS + pacmod telemetry
      └ obstacle/ped data from fusion topic and/or predictor tensor depending on prediction_source params
      └ publishes pacmod actuator commands (/pacmod/global_cmd, steering_cmd, …)

parallel controller paths (other launches)
----------------------------------------
  stanley_controller / AdaptStraightPath / PedestrianAwarePath
      └ overlapping pacmod subscriptions (enabled, vehicle_speed_rpt) and overlapping pacmod command publishers

  AdaptStraightPath
      └ also listens /safety_decision (would require a working safety node)

supporting stacks
-----------------
  high_level_command ← fusion + sign + motion subscribers → decision publishers (see adapt_high_level_command.py)
  pacmod2_driver(+component) ⇄ SocketCAN ⇄ vehicle bus
  safety_controller would sit in same launch graph once implemented (currently [ ])

third-party sensing front-ends feeding adapt
---------------------------------------------
  ouster (/ouster/points default for lidar_processing), ZED/oak/etc image topics feeding rgbd/detector pipelines, GNSS receivers publishing /navsatfix (+ driver-specific INS topics).
