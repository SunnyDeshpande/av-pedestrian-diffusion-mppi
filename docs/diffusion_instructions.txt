================================================================================
DIFFUSION PEDESTRIAN-MOTION PREDICTOR
Detailed instructions to train and deploy a conditional-diffusion
model that replaces the current heuristic predictor in this repo
(github.com/het915/cs_588_g10, "the adapt repo").

This file is self-contained. After `git pull`, a fresh engineer (or
fresh Claude session) should be able to read this top-to-bottom and
execute every step without prior conversation context.

================================================================================
0. TL;DR / QUICK START
================================================================================

Goal: replace the constant-velocity pedestrian predictor in
src/yolo_person_detector/yolo_person_detector/pedestrian_behaviour_predictor.py
with a learned, multi-modal diffusion model. Detection (YOLOv11 +
LiDAR/RGBD fusion) is unchanged. MPPI's obstacle tensor upgrades from
(M, 2) static to (M, H, 2) time-indexed in the final phase.

Dataset path: pretrain on Argoverse 2 Motion Forecasting, finetune on
~4-6 hours of self-collected GEM rosbags from the highbay.

Model: MID-style (Motion Indeterminacy Diffusion) Transformer
denoiser, ~3-4 M params, DDPM-100 train / DDIM-10 inference, K=20
samples per pedestrian, 5 s horizon at dt=0.25 s (20 steps).

Compute: training on the RTX 3060 Laptop (~6 GB usable VRAM),
~12 h pretrain + ~1 h finetune. Inference on the vehicle GPU at
< 30 ms per tick (10 Hz control loop).

End-to-end effort: ~5-6 weeks single-engineer.

Locked-in decisions (do not re-litigate):
  - Argoverse 2 pretrain + GEM finetune (NOT public-only, NOT
    self-collected-only).
  - K=20 multi-modal samples, collapse to "best mode" for runtime.
  - Prediction-only scope; YOLOv11 + LiDAR/RGBD fusion stay as the
    detection front-end.
  - 5 s horizon, 20 steps, dt=0.25 s — exact match to today's RViz
    marker so the existing visualization keeps working.


================================================================================
1. WHY THIS WORK EXISTS
================================================================================

The current predictor in
src/yolo_person_detector/yolo_person_detector/pedestrian_behaviour_predictor.py
does the following:

  Subscribed:
    /fusion_pedestrian_position  std_msgs/Int32MultiArray
        flat polar list [dist_m, dir_deg, dist_m, dir_deg, ...]
        relative to base_link; dist in integer meters,
        dir in integer degrees CCW from +x (vehicle right);
        90 deg = forward, 180 = left, 270 = back.
    /vehicle_rpt                 pacmod2_msgs/VehicleSpeedRpt
        used as a TTC gate (skip if speed < 0.01 m/s).

  Internal state (per track id):
    - Greedy 2-meter Euclidean association (max 2.0 m).
    - History buffer up to 100 points.
    - Track deletion after 10 unmatched frames.
    - Smoothing pipeline:
        (a) median spike filter, threshold 0.5 m;
        (b) 7-point moving average;
        (c) 15-point velocity estimate (non-uniform-dt, 2-sigma
            outlier rejection);
        (d) z-flatten to ground plane (z = 0);
        (e) EMA position update with alpha = 0.6.

  Prediction:
    Pure constant-velocity linear extrapolation. Horizon 5.0 s,
    20 points (i.e. dt = 0.25 s). Velocity clamped to 3.0 m/s.
    Path clipped at 15 m radius.

  Published:
    /person_prediction      visualization_msgs/Marker  (LINE_STRIP,
                                                       base_link)
    /pedestrian_motion      geometry_msgs/Twist        (selected ped)
    /pedestrian_ttc         std_msgs/Float64           (seconds)

This is brittle for two reasons:
  - Pedestrian motion is multi-modal — they slow, accelerate, change
    direction, hesitate at curbs. Constant velocity is wrong most of
    the time and especially wrong at the moments that matter (path
    intersections within 1-3 s).
  - MPPI currently consumes /fusion_pedestrian_position directly as
    static (M, 2) per-tick obstacles (see adapt_mppi_node.py and
    mppi.py), which means it cannot reason about *future* pedestrian
    occupancy; it only reacts to the current snapshot.

The design doc (docs/adapt_design.md, "Aspirational research track")
already commits to upgrading MPPI's obstacle term to a time-indexed
(M, H, 2) tensor whenever a trajectory predictor exists. This work
delivers that predictor and does the MPPI upgrade in Phase D.


================================================================================
2. WHAT WILL CHANGE WHEN THIS LANDS
================================================================================

A new ROS 2 package src/pedestrian_diffusion_predictor/ is added
sibling to src/yolo_person_detector/. Its `infer_node` is a drop-in
replacement for `pedestrian_behaviour_predictor` — same subscribed
topics, same published topics, same message types, same coordinate
frames. The downstream node adapt_high_level_command.py does not
change.

infer_node also publishes one new topic, /pedestrian_predictions_tensor
(std_msgs/Float32MultiArray with shape (M, 20, 2)), consumed only by
MPPI in Phase D.

In Phase D, two files in src/vehicle_drivers/mppi_controller/ are
modified to consume the new tensor and time-index the obstacle cost.

Detection (YOLOv11 + LiDAR/RGBD fusion in
src/adapt_full/adapt_full/) is NOT changed.


================================================================================
3. DROP-IN I/O CONTRACT
================================================================================

The new node MUST satisfy this contract on Day 1. Do not change topic
names, message types, frames, units, or shapes.

Subscribed:
  /fusion_pedestrian_position    std_msgs/Int32MultiArray
      Format: flat list of int pairs [dist_m, dir_deg, ...] in
      base_link polar; len(data) is even, M = len(data) // 2.
      Convert to Cartesian:
          theta_rad = (dir_deg - 90.0) * pi / 180.0   # 90 deg = +x
          x = dist * cos(theta_rad)
          y = dist * sin(theta_rad)
      (Equivalent to the conversion already in
       pedestrian_behaviour_predictor.py — copy that code verbatim.)

  /vehicle_rpt                   pacmod2_msgs/VehicleSpeedRpt
      Use vehicle_speed (m/s) for the TTC gate
      (skip TTC if speed < 0.01 m/s).

Published:
  /person_prediction             visualization_msgs/Marker
      LINE_STRIP type; ns = "person_prediction"; frame_id = "base_link".
      Points: 20 entries, each Point(x, y, z=0). The 20 points are
      the *selected* pedestrian's predicted trajectory at
      dt = 0.25 s, starting at t=0.25 s and ending at t=5.0 s.
      Color and lifetime: copy from the existing predictor.

  /pedestrian_motion             geometry_msgs/Twist
      linear.x = selected ped current x in base_link (m)
      linear.y = selected ped current y in base_link (m)
      linear.z = 0
      angular.* = 0
      "Selected" = the pedestrian with the lowest TTC; if no
      collision, the closest one in Euclidean distance.

  /pedestrian_ttc                std_msgs/Float64
      Time-to-collision in seconds for the selected ped, or
      math.inf if no collision predicted.
      Algorithm:
          for each predicted point i (time t_i = (i+1) * dt = 0.25, 0.5, ...):
              ego_at_t = (vehicle_speed * t_i, 0)   # forward straight-line
              dist = ||ped_pred[i] - ego_at_t||
              if dist <= 1.0 (collision_distance_threshold):
                  return t_i
          return inf

New (added by this work):
  /pedestrian_predictions_tensor std_msgs/Float32MultiArray
      Layout (matches Float32MultiArray.layout convention):
          dim[0].label = "M",  dim[0].size = M, dim[0].stride = M*20*2
          dim[1].label = "H",  dim[1].size = 20, dim[1].stride = 20*2
          dim[2].label = "xy", dim[2].size = 2, dim[2].stride = 2
          data = flat float32 array of length M * 20 * 2
      Per pedestrian m, point h: (data[m*40 + h*2], data[m*40 + h*2 + 1])
      in base_link Cartesian (m). The H trajectory is the chosen
      "best mode" of the K=20 samples (see Section 5.5).

Coordinate frames:
  All Cartesian outputs are base_link. x = forward, y = left, z = up.
  No tf lookups needed inside the new node — fusion is already in
  base_link polar.

================================================================================
5. MODEL — MID-STYLE TRANSFORMER DENOISER
================================================================================

The model is intentionally small, well-cited, and operates on
trajectory tensors only. It does NOT consume HD maps (the GEM e4
has none) or social-graph context (Day-1 simplicity).

5.1 Inputs (per pedestrian, per inference)
------------------------------------------
  history:   tensor (T_hist=20, 4) of [x, y, vx, vy]
             - 20 steps of past observations at dt = 0.25 s
               (i.e. last 5 seconds of history)
             - x, y in ego frame at t=0 (subtract current ego pose,
               rotate so ego heading is +x)
             - vx, vy = first-difference of x, y divided by dt; for
               the first sample, set vx=vy=0
             - if a track is younger than 20 frames, left-pad with
               zeros and provide a presence mask
  hist_mask: tensor (T_hist=20,) of {0, 1}, 1 = real observation
  ego_vel:   tensor (2,) of [v_lin, v_ang] from /vehicle_rpt
             (v_ang is 0 if not available)
  noise_t:   integer in [0, 99] — current diffusion timestep during
             training, current denoising step during DDIM inference

5.2 Outputs
-----------
  noise_pred: tensor (T_fut=20, 2) — the predicted noise epsilon
              the denoiser thinks was added to the clean future
              trajectory. (Standard DDPM eps-prediction.)

  After full denoising (T_fut x K samples per pedestrian):
  futures:    tensor (K=20, T_fut=20, 2) — K candidate Cartesian
              trajectories in ego frame, each starting at t=0.25 s
              and going to t=5.0 s.

5.3 Architecture
----------------
  Embedding:
    history -> Linear(4 -> 128) -> add positional embedding
                                    (learned, 20 entries)
                                -> add presence-mask gating
    ego_vel -> Linear(2 -> 128)  (broadcast/added to every history token)
    noise_t -> SinusoidalEmbed(128) -> Linear(128 -> 128)
                                    (used for AdaLN modulation)

  Encoder:
    4 stacked TransformerEncoderLayer blocks
      - num_heads = 4
      - d_model = 128
      - dim_feedforward = 256
      - dropout = 0.1
      - AdaLN: scale and shift parameters produced from
               noise_t embedding (concat ego_vel embedding)

  Decoder head:
    Cross-attention from a fixed (T_fut=20, 128) learned query
    sequence onto the encoder output, then 2-layer MLP per token to
    produce (T_fut=20, 2) eps-prediction.

  Total parameters: ~3-4 M. Checkpoint size ~15 MB. Profile with
    sum(p.numel() for p in model.parameters() if p.requires_grad)

5.4 Diffusion process
---------------------
  Forward (training):
    Sample t ~ Uniform{0, ..., 99}.
    Sample eps ~ N(0, I) of shape (T_fut, 2).
    Build alpha_bar from cosine schedule (Nichol & Dhariwal, 2021):
      f(t) = cos((t/T + 0.008) / 1.008 * pi/2)^2
      alpha_bar_t = f(t) / f(0)
    Noised target:
      y_t = sqrt(alpha_bar_t) * y_clean + sqrt(1 - alpha_bar_t) * eps
    Predict eps_hat from history + ego_vel + t.
    Loss = MSE(eps_hat, eps).

  Reverse (inference, DDIM, deterministic):
    Choose 10 timesteps spaced linearly from 99 to 0
      (e.g. tau = [99, 88, 77, 66, 55, 44, 33, 22, 11, 0]).
    Initialize y_99 ~ N(0, I) of shape (K, T_fut, 2).
      Use a fixed per-tick seed (e.g. derived from track_id) for
      stability — see Section 13 (Risks).
    For each consecutive pair (tau_i, tau_{i-1}):
      eps_hat = model(history, ego_vel, y_tau_i, tau_i)
      y0_pred = (y_tau_i - sqrt(1 - alpha_bar_tau_i) * eps_hat)
                / sqrt(alpha_bar_tau_i)
      y_tau_{i-1} = sqrt(alpha_bar_tau_{i-1}) * y0_pred
                  + sqrt(1 - alpha_bar_tau_{i-1}) * eps_hat
    Output y_0 of shape (K, T_fut, 2).

  K-samples-per-pedestrian batching:
    Tile history along a leading "K" dim, run K denoising chains in
    parallel as a single batch of size M*K. Model itself is
    stateless wrt K — it is identical to single-sample inference.

5.5 Best-mode selection (runtime)
---------------------------------
  Goal: from K=20 samples, pick ONE trajectory per pedestrian to
  populate /person_prediction (Marker), /pedestrian_motion (Twist),
  /pedestrian_ttc (Float64), and /pedestrian_predictions_tensor
  (one slice per pedestrian).

  Algorithm: "closest-to-mean" with sticky temporal selection.

    mean_path = futures.mean(dim=0)            # (T_fut, 2)
    cost_per_sample = ((futures - mean_path)**2).sum(dim=(1,2))  # (K,)
    candidate = futures[argmin(cost_per_sample)]

    sticky:
      if track has previous chosen mode 'prev_idx':
          cost_prev = cost_per_sample[prev_idx]
          cost_cand = cost_per_sample.min()
          if cost_prev > 1.5 * cost_cand for 3+ consecutive ticks:
              switch to candidate; reset counter
          else:
              keep prev_idx
      else:
          set prev_idx = argmin(cost_per_sample)

  This avoids frame-to-frame mode flapping that would make TTC
  jitter wildly. Tune the 1.5x ratio if A/B replay shows over- or
  under-switching.

5.6 Why MID and not Trajectron++ / AgentFormer / SocialDiffusion
-----------------------------------------------------------------
  - GEM e4 has no HD map -> map-conditioned models add params and
    plumbing for no benefit.
  - Day-1 per-agent independence is fine; the highbay scenario rarely
    has dense pedestrian groups.
  - MID has the simplest published implementation, the smallest
    parameter count for the target metric, and no irregular
    components (no graph attention over agent counts that vary
    frame-to-frame).
  - We can later swap the backbone for SocialDiffusion if A/B replay
    flags collision-rate problems with M > 1.

6. DATASET — STAGE 1: ARGOVERSE 2 PRETRAIN
================================================================================

Source: Argoverse 2 Motion Forecasting Dataset
  - License: CC-BY 4.0 (academic use OK; cite in package.xml)
  - Download: https://www.argoverse.org/av2.html#forecasting-link
    requires the official Python package `av2` (pip install av2)
  - Size: ~60 GB compressed across train/val/test
  - Sample rate: 10 Hz
  - Window: 11 s total per scenario (5 s history + 6 s future);
    we use 5 s history + 5 s future
  - Vehicle frame, 2D xy — exact modality match to /fusion_pedestrian_position

Expected pedestrian sequences after filtering: ~150,000.

6.1 Download
------------
  pip install av2 boto3
  s5cmd cp 's3://argoverse/datasets/av2/motion-forecasting/*' \
         data/av2/                                   # ~60 GB
  # Or use the official downloader from the av2 docs.

6.2 Preprocess (one-time)
-------------------------
  Script: pedestrian_diffusion_predictor/scripts/preprocess_av2.py

  For each scenario:
    1. Load with av2.datasets.motion_forecasting.scenario_serialization
    2. Identify the focal-track frame at t=0 (the AV at the moment
       of prediction).
    3. For every track with category == 'pedestrian':
         - Skip if track has < 5 s of history before t=0.
         - Skip if track has < 5 s of future after t=0.
         - Skip if any inter-step gap > 0.2 s.
         - Extract:
             history_xy_world = positions at t = -4.75, -4.50, ..., 0.0
                                                 (20 points, dt=0.25)
             future_xy_world  = positions at t = 0.25, 0.50, ..., 5.0
                                                 (20 points)
       (AV2 is 10 Hz; downsample by 2 to hit dt = 0.25 s, OR use
        every frame and keep T_hist = T_fut = 50 — pick one
        consistently. We pick downsample-by-2 to match the deployed
        contract exactly.)
    4. Normalize to ego (AV) frame at t=0:
         T = ego_pose_world_at_t0       # 4x4 SE(2) lifted to SE(3)
         history_xy_ego = (T^-1 * history_xy_world)[:, :2]
         future_xy_ego  = (T^-1 * future_xy_world )[:, :2]
    5. Compute history velocities by first-difference, dt = 0.25 s.
    6. Compute ego velocities at t=0 from AV's own track.
    7. Save shard.

  Output format (Parquet, one row per pedestrian sequence):
    history       float32  shape (20, 4)   [x, y, vx, vy]
    history_mask  uint8    shape (20,)     all 1s after filtering
    future        float32  shape (20, 2)   [x, y]
    ego_vel       float32  shape (2,)      [v_lin, v_ang]

  Sharding: ~10 MB per shard; ~2,000 shards total. Use pyarrow
  with row-group size 8192. The whole preprocessed set fits in
  ~6 GB on disk.

6.3 Augmentation (applied at dataloader time, not preprocess)
-------------------------------------------------------------
  - Random rotation +/- 15 degrees applied to BOTH history_xy and
    future_xy together (preserves trajectory consistency).
  - Random history dropout: zero out a random 0-20% of history
    steps and clear the mask bit. Simulates detection gaps.
  - Random translation jitter: +/- 0.1 m per coordinate at t=0.
  - NO horizontal flip — pedestrian motion is not symmetric in the
    presence of a road / sidewalk semantic.

7. DATASET — STAGE 2: GEM ROSBAG FINETUNE
================================================================================

Distribution shift between AV2 (urban crosswalks, mostly) and the
UIUC highbay (research lab parking lot, confederate pedestrians,
slower speeds, more random paths) is the biggest risk to deployment
ADE. Stage 2 finetune is the primary mitigation.

7.1 Recording protocol
----------------------
  Vehicle: Polaris GEM e4 (E4) at the highbay.
  Sensors active: Ouster OS1-128, Livox HAP, OAK-D LR, Septentrio
  GNSS, PACMod2.
  Bring up sensors with:
    bash run_tmux.sh
  (or `ros2 launch basic_launch sensor_init.launch.py vehicle_name:=e4`)

  Topics to record:
    /fusion_pedestrian_position
    /lidar_pedestrian_position
    /rgbd_pedestrian_position
    /vehicle_rpt
    /pacmod/vehicle_speed_rpt
    /tf  /tf_static
    /clock                                # if simulated time

  Command:
    ros2 bag record -o data/gem_bags/<date>_<scenario>_<n> \
      /fusion_pedestrian_position /vehicle_rpt /tf /tf_static

  Total target: 4-6 hours.
    - Crossing: ~150 events (~2 hours of bag).
        Confederate pedestrians cross perpendicular to vehicle path
        at varying standoff distances (5, 10, 15, 20 m) and walking
        speeds (slow stroll, normal, brisk).
    - Parallel-walk: ~80 events (~1.5 hours).
        Pedestrian walks alongside the vehicle, same direction or
        opposite, at varying lateral offset.
    - Stationary: ~50 events (~30 minutes).
        Pedestrian stands in or near the ego path; tests the "no
        motion" mode.
    - Group: ~20 events (~30 minutes).
        2-4 confederates walk together; tests M > 1 batching even
        if Day-1 model treats agents independently.

  Vehicle speed during recording: 1-4 m/s (matches deployment-mode
  desired_speed in adapt_mppi_launch.py). Drive manually; engaging
  PACMod is NOT necessary.

  Diversity discipline:
    - Two recording days, different lighting.
    - Multiple confederate clothing colors (YOLO already detects
      arbitrary humans, but this guards against bias).
    - Vary intersection / standoff geometry every 3-5 events.

7.2 Auto-labeling
-----------------
  No manual annotation. The recorded /fusion_pedestrian_position
  IS the label, because the existing fusion pipeline is the ground
  truth we are training against.

  Script: pedestrian_diffusion_predictor/scripts/extract_gem_windows.py

  For each bag:
    1. Replay /fusion_pedestrian_position offline (rosbag2 reader).
    2. Run the same tracker as the existing predictor (greedy
       2 m association, smoothing). Track histories live in the
       tracker's `tracks` dict.
    3. For each track id, every time the history reaches >= 20
       observed frames AND has at least 20 frames of future
       observation in the bag, emit one (history, future, ego_vel)
       window.
    4. Use /vehicle_rpt at the window's t=0 for ego_vel.
    5. Save to the same Parquet schema as Section 6.

  Expected windows: ~5,000-10,000 from 4-6 hours of bag, depending
  on event density. A small set is fine — finetune is only 20 epochs.

7.3 Holdout split
-----------------
  Reserve ~20% of recorded bags (different recording day, different
  scenarios) as a UIUC validation set. Compute minFDE-20 / minADE-20
  on this set after finetune; this is the deployment metric.

9. CODE SKELETONS

9.1 ddpm.py
-----------
  import math, torch

  class CosineSchedule:
      def __init__(self, T=100, s=0.008):
          self.T = T
          steps = torch.arange(T+1, dtype=torch.float64)
          f = torch.cos((steps/T + s) / (1+s) * math.pi/2)**2
          self.alpha_bar = (f / f[0]).float()              # (T+1,)
          self.alphas    = self.alpha_bar[1:] / self.alpha_bar[:-1]
          self.betas     = (1.0 - self.alphas).clamp(0, 0.999)

      def q_sample(self, x0, t, eps):
          ab = self.alpha_bar[t+1].view(-1, 1, 1)
          return ab.sqrt() * x0 + (1 - ab).sqrt() * eps

      def ddim_step(self, xt, eps_hat, t_now, t_next):
          ab_now  = self.alpha_bar[t_now + 1].view(-1, 1, 1)
          ab_next = self.alpha_bar[t_next + 1].view(-1, 1, 1)
          x0_pred = (xt - (1 - ab_now).sqrt() * eps_hat) / ab_now.sqrt()
          return ab_next.sqrt() * x0_pred + (1 - ab_next).sqrt() * eps_hat

9.2 model.py (sketch)
---------------------
  import torch, torch.nn as nn

  class TrajectoryDenoiser(nn.Module):
      def __init__(self, d=128, T_hist=20, T_fut=20):
          super().__init__()
          self.hist_in = nn.Linear(4, d)
          self.hist_pos = nn.Parameter(torch.randn(T_hist, d) * 0.02)
          self.ego_in = nn.Linear(2, d)
          self.t_embed = nn.Sequential(
              SinusoidalPosEmb(d), nn.Linear(d, d), nn.SiLU(),
              nn.Linear(d, d))
          enc_layer = nn.TransformerEncoderLayer(
              d_model=d, nhead=4, dim_feedforward=256,
              dropout=0.1, batch_first=True, norm_first=True)
          self.enc = nn.TransformerEncoder(enc_layer, num_layers=4)
          self.queries = nn.Parameter(torch.randn(T_fut, d) * 0.02)
          self.dec = nn.MultiheadAttention(d, 4, batch_first=True)
          self.head = nn.Sequential(
              nn.Linear(d, d), nn.SiLU(), nn.Linear(d, 2))

      def forward(self, hist, hist_mask, ego_vel, t):
          # hist:(B, T_hist, 4)  ego_vel:(B, 2)  t:(B,) long
          B = hist.shape[0]
          h = self.hist_in(hist) + self.hist_pos
          ego = self.ego_in(ego_vel).unsqueeze(1)              # (B,1,d)
          tt = self.t_embed(t).unsqueeze(1)                    # (B,1,d)
          tokens = h + ego + tt                                # AdaLN-lite
          # mask invalid history steps
          src_mask = (hist_mask == 0)                           # (B, T_hist)
          z = self.enc(tokens, src_key_padding_mask=src_mask)
          q = self.queries.unsqueeze(0).expand(B, -1, -1)
          y, _ = self.dec(q, z, z)
          return self.head(y)                                  # (B, T_fut, 2)

9.5 eval.py (skeleton)
----------------------
  Loads a checkpoint, generates K=20 samples per validation
  pedestrian, computes:
    minADE_K = mean over peds of min over K of mean over T of
               ||y_kt - y_gt_t||_2
    minFDE_K = mean over peds of min over K of ||y_kT - y_gt_T||_2
    miss_rate@2m = fraction of peds for which all K samples have
                   FDE > 2.0 m
  Run command:
    python -m pedestrian_diffusion_predictor.eval \
        --ckpt models/diffusion/.../ema_final.pt \

    python -m pedestrian_diffusion_predictor.eval \
        --benchmark --M 8 --K 20 --device cuda:0
  Reports forward-pass latency in ms.

9.6 infer_node.py (ROS 2 node skeleton)
---------------------------------------
  Inherits from rclpy.node.Node. Same logger / params layout as
  pedestrian_behaviour_predictor.py. Key differences:

    - Holds a torch.nn.Module on `self.device` (declared param,
      default 'cuda:0' if available else 'cpu').
    - Loads checkpoint at startup from a `weights` parameter.
    - Holds the same Tracker instance as the old node (copied
      module).

  Tick callback (driven by /fusion_pedestrian_position arrival):
    1. Decode polar -> base_link Cartesian, pass to tracker.
    2. For every active track:
         hist = tracker.get_history(track_id, T_hist=20)
                                              # pad + mask if short
         skip if presence count < 5 (model is unreliable on
                                      <1.25 s of history).
    3. Build batch: (M, T_hist, 4), (M, T_hist), (M, 2).
    4. Tile by K to get (M*K, T_hist, 4) etc.
    5. Run DDIM-10 inference -> futures of shape (M*K, T_fut, 2).
       Reshape to (M, K, T_fut, 2).
    6. Sticky-best-mode select per track -> (M, T_fut, 2).
    7. Publish:
         /pedestrian_predictions_tensor (M, 20, 2)
         /person_prediction (Marker for selected ped)
         /pedestrian_motion (Twist for selected ped)
         /pedestrian_ttc    (Float64)
    8. Latency timer: log a warning if cycle > 80 ms.

  Selection of "selected ped" matches the old node:
    - if any track has finite TTC: smallest TTC.
    - else: smallest current Euclidean distance from base_link origin.

13. RISK REGISTER & MITIGATIONS
================================================================================

R1. Distribution shift (AV2 -> UIUC).
    Severity: high.
    Detection: UIUC minFDE-20 > 1.5x AV2 val minFDE-20.
    Mitigation: Stage-2 finetune on GEM bags. If still failing,
    extend recording to 12-15 hours (~600 events). Last resort:
    retrain AV2 with augmentation that randomly rescales
    pedestrian speed by 0.5-1.5x to reduce dataset bias toward
    crosswalk speeds.

R2. Inference latency under M > 1.
    Severity: medium.
    Detection: bench_latency.py reports > 50 ms at M=8, K=20.
    Mitigation: drop DDIM steps, then d_model, then K. Last
    resort: distill to a 1-step student (BCDiff-style) — adds a
    week of work.

R3. Mode-flip jitter.
    Severity: medium.
    Detection: TTC oscillates frame-to-frame on a stationary or
    slowly-moving pedestrian.
    Mitigation: sticky-mode selection (Section 5.5). Tune the
    1.5x switch threshold based on observed flapping.

R4. MPPI cost numerics with time-indexed obstacles.
    Severity: medium.
    Detection: oscillation in steering or accel commands when a
    pedestrian is in the path; non-monotone cost surface.
    Mitigation: keep the soft Gaussian repulsion; do not introduce
    hard step at clearance radius. If the issue persists, add a
    short low-pass filter on commands (cutoff ~3 Hz) — the
    existing filter/cutoff parameter is already in the launch
    file.

R5. DDIM frame-to-frame stochasticity.
    Severity: low.
    Detection: predicted trajectory wiggles between adjacent
    inference calls even when nothing changed.
    Mitigation: seed the initial noise per (track_id, tick_index)
    deterministically, OR average two seeds, OR (cheapest) keep
    the previous tick's noise and re-run DDIM with one extra step
    to "re-denoise" with a small kick.

R6. Dataset license.
    Severity: low.
    Detection: legal review.
    Mitigation: AV2 is CC-BY 4.0; cite Argoverse 2 in
    pedestrian_diffusion_predictor/package.xml description and in
    docs/adapt_design.md (one-line credit).

R7. Repo hygiene.
    Severity: medium (operational).
    Mitigation: BEFORE the first training run, add to
    cs_588_g10/.gitignore:
        data/
        models/
        *.pt
        *.ckpt
        logs/
    Verify with `git status` after staging that no .pt/.ckpt or
    parquet files appear.

R8. Conda env mismatch.
    Severity: low.
    Detection: `python` reports the wrong path.
    Mitigation: training uses the `adapt` env (py3.12, torch
    2.5.1+cu121); CPU-only dev uses `cs588`. The conda env policy
    is documented in CLAUDE.md — do NOT install with apt or into
    ~/.local.

R9. Vehicle GPU mismatch.
    Severity: medium.
    Detection: torch.cuda.get_device_name(0) on the vehicle does
    not match the laptop.
    Mitigation: re-check latency on the vehicle GPU at deployment
    time. If too slow, use the runtime fallbacks in R2.
